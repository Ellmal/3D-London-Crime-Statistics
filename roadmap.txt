# roadmap.txt — London Crime 3D Map Pulse Explorer

## Purpose

The project uses real London street-level crime statistics, organised by month.

The goal is to build toward an interactive London Crime Pulse Explorer that helps users explore reported street-level crime across London by:

- month
- location
- LSOA
- crime type
- crime intensity
- spatial aggregation
- 3D hexagon (hexbin) visualisation and timelapse behaviour

The final visual target is a 3D hexbin map: crime points binned into hexagonal
cells, with height and colour representing crime intensity, on a dark basemap
with a tilted camera. This is built with PyDeck, using deck.gl rendering.

The project should still be built as a learning ladder.

Do not jump directly to the final application.

The correct learning progression is:

Raw monthly files
→ file inventory
→ schema profiling
→ loading
→ cleaning
→ validation
→ processed data
→ basic data explorer + interactive point map
→ spatial aggregation
→ 2D intensity validation map
→ 3D hexagon extrusion map (primary visual)
→ filters and controls
→ monthly slider / timelapse
→ MVP
→ polish

---

# Key Learning Rule

Learn the process first.

Early stages should focus on:

- how real files are loaded
- how the schema is understood
- how bad data is detected
- how raw data becomes clean data
- how point data becomes aggregated data
- how aggregated data becomes a map
- how 2D intensity works before 3D height is added

Only after those steps are stable should the project add:

- 3D hexagons
- advanced filters
- time sliders
- deployment
- AI features

# Important Performance Rule

While testing and learning, load only May 2025.

Use only:

data/raw/2025-05/

This is a deliberate learning/performance decision.

Do not process the full year until the May 2025 pipeline works end-to-end.

The correct expansion path is:

1. Build and test everything with May 2025 only.
2. Then add one more month.
3. Then process all available months.
4. Then build monthly timelapse logic.

This avoids slow debugging and keeps each stage understandable.

# Expected Crime Data Columns

The UK police street-level crime CSV files usually contain:

- Crime ID
- Month
- Reported by
- Falls within
- Longitude
- Latitude
- Location
- LSOA code
- LSOA name
- Crime type
- Last outcome category
- Context

Important notes:

- `Crime ID` can be missing.
- `Month` is usually formatted like `2025-05`.
- `Longitude` and `Latitude` are needed for point maps.
- `LSOA code` and `LSOA name` are needed for local-area aggregation.
- `Crime type` is needed for grouping and filtering.
- Some records may have missing coordinates.
- Some records may fall outside the expected London display area.

---

# Suggested London Bounding Box

Use this only as an early visual filter, not as a permanent data deletion rule.

Suggested approximate bounding box:

- longitude between -0.55 and 0.35
- latitude between 51.25 and 51.75

Create a boolean field:

- is_within_london_bbox

Do not delete out-of-bounds records too early. Flag them first.

# Data Architecture

## Raw Layer

Location:

data/raw/

Purpose:

Stores untouched monthly CSV files.

Rules:

- Never edit raw files manually.
- Never overwrite raw files from the pipeline.
- Keep each month in its own folder.
- Keep police-force files separate in raw storage.
- Use code to load and combine files.

---

## Processed Layer

Location:

data/processed/

Purpose:

Stores cleaned row-level data.

Possible outputs:

- data/processed/london_crime_clean_2025-05.parquet
- data/processed/london_crime_clean_all.parquet

Rules:

- Columns should be standardized.
- Coordinates should be numeric.
- Month should be parsed.
- LSOA fields should be preserved.
- Crime type should be preserved.
- Source file metadata should be preserved.
- A generated row_id should exist because Crime ID may be missing.

---

## Visualisation Layer

Location:

data/viz/

Purpose:

Stores visual-ready aggregated datasets.

Possible outputs:

- data/viz/crime_by_month_type.parquet
- data/viz/crime_by_lsoa_month_type.parquet
- data/viz/crime_grid_month_type.parquet
- data/viz/crime_grid_2d_intensity.parquet
- data/viz/crime_hex_3d_2025-05.parquet (3D hexbin visual-ready data)

Rules:

- Visual-ready data should be smaller than processed data.
- Expensive calculations should happen before the Streamlit app loads.
- The app should mostly read prepared data, not do heavy transformations each time.

# Mini 4 — Cleaning Pipeline

## Goal

Standardize and clean the May 2025 real crime data.

## Input

May 2025 raw CSV files from:

data/raw/2025-05/

## Cleaning Tasks

1. Standardize column names.

Example mapping:

- Crime ID → crime_id
- Month → month
- Reported by → reported_by
- Falls within → falls_within
- Longitude → longitude
- Latitude → latitude
- Location → location
- LSOA code → lsoa_code
- LSOA name → lsoa_name
- Crime type → crime_type
- Last outcome category → last_outcome_category
- Context → context

2. Convert longitude and latitude to numeric.

3. Flag coordinate status.

Create useful fields such as:

- has_coordinates
- has_valid_coordinates
- is_within_london_bbox

4. Validate coordinate ranges.

Rules:

- latitude must be between -90 and 90
- longitude must be between -180 and 180

5. Apply optional London bounding-box flag.

Use:

- longitude between -0.55 and 0.35
- latitude between 51.25 and 51.75

Do not delete out-of-bounds rows in the master cleaned dataset unless explicitly building map-ready data.

6. Parse month.

Create:

- month_start_date
- year
- month_number
- month_label

7. Derive data source name.

Build a short uppercase acronym from the first letter of each word in
`falls_within`:

- City of London Police → COLP
- Metropolitan Police Service → MPS

Create:

- data_source_name

8. Preserve LSOA fields.

Keep:

- lsoa_code
- lsoa_name

9. Preserve crime type.

Keep:

- crime_type

10. Create generated row id.

Because crime_id can be missing, create:

- row_id

Suggested row_id logic:

LCS / data_source_name / month / source_row_number

11. Preserve source metadata.

Keep:

- source_file
- source_month_folder
- source_row_number

## Suggested File

src/cleaning/clean_crime_data.py

## Output

data/processed/london_crime_clean_2025-05.parquet

## Success Criteria

- Cleaned Parquet output exists.
- Standardized column names exist.
- Coordinates are numeric where available.
- Coordinate flags exist.
- Month fields exist.
- Data source name is derived from falls_within.
- LSOA fields exist.
- Crime type exists.
- Source metadata exists.
- Missing Crime ID does not break uniqueness.

# Mini 8 — Spatial Aggregation

## Goal

Move from raw points to summarized spatial activity.

This is a major learning step.

Do this before 3D.

## Why This Matters

Raw crime points become cluttered and slow.

A Pulse Explorer should summarize crime intensity by area.

## First Simple Grid Approach

Before using advanced geospatial tools, create a basic grid using rounded coordinates.

Example:

- lat_grid = round(latitude, 3)
- lon_grid = round(longitude, 3)

Then group by:

- month
- lat_grid
- lon_grid

Optional later groupings:

- month
- lat_grid
- lon_grid
- crime_type

## Aggregations

For each grid cell calculate:

- crime_count
- unique_crime_type_count
- top_crime_type
- unique_lsoa_count
- center_latitude
- center_longitude

## Input

data/processed/london_crime_clean_2025-05.parquet

Use rows where:

- has_valid_coordinates = True
- is_within_london_bbox = True

## Output

data/viz/crime_grid_summary_2025-05.parquet

## Suggested File

src/transformation/aggregate_crime_grid.py

## Success Criteria

- Aggregated dataset exists.
- Number of rows is smaller than raw point data.
- Each row represents one spatial cell.
- Aggregated crime counts are correct.
- Dataset can be used for validation, summary stats, the 2D intensity check,
  and the later 3D hexbin map.

# Mini 10 — 3D Hexagon Extrusion Map

## Goal

Build the primary visual of the project: a 3D hexbin map of London crime with
hexagonal cells, height and colour for intensity, a dark basemap, and a tilted
camera.

## Primary Layer

- PyDeck `HexagonLayer` with `extruded=True`.

`HexagonLayer` bins points into hexagons on the GPU and extrudes each hexagon
based on point density/count.

## Input

Primary input: cleaned crime points (filtered by month, and by crime type when
filtering), so `HexagonLayer` can bin them directly:

data/processed/london_crime_clean_2025-05.parquet

Use rows where:

- has_valid_coordinates = True
- is_within_london_bbox = True

The grid summary from Mini 8 stays useful for validation and summary stats.
Optionally precompute a hex visual-ready dataset:

data/viz/crime_hex_3d_2025-05.parquet

## Visual Encoding

- position = longitude / latitude (binned into hexagons)
- height = crime count / density per hexagon
- color = crime count / density (sequential scale)
- tooltip = crime count (and breakdown if available)

## Required Visual Spec

These decisions define the visual style:

- dark basemap (e.g. Carto Dark / a dark map style)
- tilted camera: pitch ~ 45–60°, with a sensible bearing
- height scaling using log or sqrt so one hotspot does not dominate
  - examples: log(count + 1) * k, or sqrt(count) * k
- a sequential colour scale (e.g. blue/low → yellow/pink/high)
- tuned hexagon `radius` / `coverage` for London's scale
- cap or scale extreme cells so the map stays readable
- hover tooltips enabled (`pickable=True`)

## Output

outputs/maps/london_crime_3d_hex_2025-05.html

## Suggested File

src/visualization/hexagon_3d_map.py

## Success Criteria

- Map shows readable 3D hexagon columns.
- Height and colour represent crime intensity.
- Dark basemap and tilted camera give a clear 3D perspective.
- User can hover and see values.
- One extreme area does not destroy readability.
- This works for May 2025 before expanding to other months.

# Mini 13 — Monthly Time Slider / Timelapse Logic

## Goal

Use monthly data to show change over one year.

## First Version

Add a month selector:

- 2025-05
- 2025-06
- ...
- 2026-04

The map updates when selected month changes.

## Second Version

Use a Streamlit select_slider.

## Third Version Later

Create a timelapse-style interaction by stepping month by month.

## Absolute vs Normalized Scaling (important for a true "pulse")

Raw crime counts differ from month to month, so taller columns can simply mean
"busier month" rather than a real change in pattern. Provide two modes:

- Absolute mode: height/colour = raw crime count
  (good for exploring a single month)
- Normalized mode: scale within each month (e.g. count / month max, or a
  z-score), or use a fixed shared scale across all months
  (good for comparing months fairly in a timelapse)

## Important Architecture

Precompute visual-ready monthly aggregations.

Do not recompute from raw data every time the slider changes.

## Input

data/viz/crime_grid_month_type.parquet

or

data/viz/crime_hex_3d_month.parquet

## Success Criteria

- User can select a month.
- The 3D hexagon map updates by month.
- Absolute and normalized scaling modes both work.
- Crime type filters still work.
- The app can show one year of change in a controlled way.
