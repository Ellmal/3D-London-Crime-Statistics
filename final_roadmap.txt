final_roadmap.txt - London Crime Pulse Explorer

Purpose
-------
This file is a compact memory of the project journey. The original
roadmap.txt is the long learning roadmap. This file records the route we
actually followed to reach the current version: a reproducible London crime
data pipeline feeding an interactive Streamlit map with a 3D hexagon crime
intensity view.


Story Summary
-------------
The project started with a simple goal: use real London street-level crime
data and gradually turn it into an interactive 3D map. Instead of jumping
straight to the final visual, we treated the work as a learning ladder. The
first concern was not the map, but trust: where the raw files live, whether
the files are named correctly, what columns they contain, and how much of the
data is usable for mapping.

The early work built that foundation. Raw monthly CSV files were organised
under data/raw by month. We created inventory and profiling steps so the
project could explain what files exist, which months are present, and whether
schemas and values look consistent. After that, the pipeline learned how to
load the monthly police files together and clean them into a standard shape.
The raw UK Police column names were converted to snake_case, coordinates were
validated, London bounding-box flags were added, month fields were derived,
and each row received a stable row_id that does not depend on Crime ID being
present.

Once the cleaned row-level data was reliable, the project moved into outputs
that are better for repeatable analysis. Cleaned Parquet files are written per
month, and when multiple months are processed a combined all-months Parquet is
also produced. Data quality reports then describe the cleaned result: row
counts, coordinate quality, missing IDs, LSOA coverage, crime type counts, and
top local areas. This made the project more than a visual experiment; it
became a small data product with checks and traceable assumptions.

The visualisation work came next in stages. We first kept raw point maps for
debugging and basic exploration. Then we built spatial aggregation using a
simple rounded lat/lon grid so crime intensity could be checked before moving
to 3D. A 2D intensity map validated that aggregation visually. After that, the
main visual layer was introduced: crime points are binned into a shared hex
grid, converted to hex cell counts, and rendered as extruded PyDeck columns on
a dark London map.

The current version is built around performance and clarity. Heavy work is
done in run_pipeline.py, not during every UI interaction. The pipeline creates
processed Parquet files plus visual-ready aggregation files in data/viz,
including crime_hex_3d_month.parquet for the 3D map. The Streamlit app loads
that small precomputed hex dataset for the primary view, then filters by month
and crime type and re-encodes colour/elevation for the current selection. Raw
points remain available as a secondary map layer for inspection, with a point
limit to keep the app responsive.

The outcome is the current London Crime Pulse Explorer: a Streamlit app that
uses real data, supports month and crime type exploration, shows summary
metrics, renders 3D hexagon crime intensity, and keeps the pipeline
reproducible from raw CSVs to visual-ready Parquet.


Technical Roadmap Covered
-------------------------

1. Project structure and data layout
   - Created a real-data project layout with data/raw, data/processed,
     data/viz, outputs/reports, outputs/maps, src, app.py, and
     run_pipeline.py.
   - Kept raw CSVs untouched and organised by YYYY-MM folders.
   - Used separate raw files for City of London Police and Metropolitan
     Police.
   - Kept outputs generated by code rather than manually editing derived data.

2. Shared configuration
   - Added src/config.py as the central place for paths and constants.
   - Added PIPELINE_MONTHS as the single source of truth for which months run.
   - Added DEFAULT_TESTING_MONTH for single-month defaults and inventory
     highlighting.
   - Added LONDON_BBOX for coordinate filtering and map framing.
   - Added shared hex map constants for radius, elevation, colour, opacity,
     and camera settings.

3. Raw file inventory
   - Built src/ingestion/list_raw_files.py.
   - Checked month folder names and expected street-level CSV filenames.
   - Detected missing months, unexpected files, month mismatches, and naming
     issues.
   - Wrote a raw file inventory report to outputs/reports.

4. Raw schema profiling
   - Built profiling so raw files could be inspected before cleaning.
   - Captured column names, missing values, row counts, and schema consistency.
   - Supported combined or separate reports depending on REPORT_OUTPUT_MODE.

5. Monthly loading
   - Built src/ingestion/load_crime_files.py to load selected months.
   - Added source metadata such as source file, source month folder, and source
     row number.
   - Allowed the same code to load one month, selected months, or all detected
     month folders.

6. Cleaning pipeline
   - Built src/cleaning/clean_crime_data.py.
   - Renamed raw police columns to snake_case using RAW_COLUMN_RENAME_MAP.
   - Converted longitude and latitude to numeric values.
   - Added coordinate flags:
     - has_coordinates
     - has_valid_coordinates
     - is_within_london_bbox
   - Derived data_source_name from the police force name.
   - Parsed month into month_start_date, year, month_number, and month_label.
   - Added stable row_id values using source metadata.

7. Processed outputs
   - Wrote cleaned monthly Parquet files to data/processed.
   - Wrote london_crime_clean_all.parquet when multiple months are processed.
   - Removed stale combined output when only one month is processed.
   - Kept processed outputs safe to regenerate from raw data.

8. Data quality validation
   - Built src/validation/validate_crime_data.py.
   - Generated monthly data quality reports.
   - Reported raw vs cleaned row counts, file counts, coordinate quality,
     missing Crime IDs, missing LSOA codes, unique row_id counts, duplicates,
     crime type distribution, and top LSOAs.
   - Used the reports to confirm that cleaned data was suitable for mapping.

9. Pipeline orchestration
   - Built run_pipeline.py as the end-to-end command.
   - The pipeline now runs:
     1. inventory raw files
     2. profile raw files
     3. load and clean raw files
     4. write processed Parquet
     5. generate data quality reports
     6. build simple spatial grid summaries
     7. build combined grid month/type data
     8. build combined hex month/type data
   - The pipeline is safe to rerun because derived reports, processed files,
     and visual summaries are overwritten from source data.

10. Simple spatial aggregation
    - Built src/transformation/aggregate_crime_grid.py.
    - Filtered to rows with valid coordinates inside the London bounding box.
    - Rounded latitude/longitude to a simple grid for early validation.
    - Aggregated crime counts, unique crime types, top crime type, and unique
      LSOA counts per grid cell.
    - Wrote grid summaries to data/viz.

11. 2D intensity validation map
    - Built src/visualization/intensity_2d_map.py.
    - Loaded grid summaries and rendered a flat PyDeck map.
    - Used colour to show crime_count intensity.
    - Used this as a check before trusting the 3D visual layer.

12. Shared hex-grid geometry
    - Built src/transformation/hex_grid.py.
    - Added local London equirectangular projection math.
    - Converted lon/lat points to axial hex coordinates.
    - Rounded fractional hex coordinates to stable hex cells.
    - Converted hex_q and hex_r back to centre lon/lat.
    - Kept this geometry separate from PyDeck so pipeline aggregation and app
      rendering use the same grid.

13. 3D hexagon aggregation and rendering
    - Added hex month/type aggregation to data/viz/crime_hex_3d_month.parquet.
    - Grouped by month, crime_type, hex_q, and hex_r.
    - Stored each hex centre longitude and latitude with the crime count.
    - Built src/visualization/hexagon_3d_map.py for colour/elevation encoding
      and PyDeck ColumnLayer rendering.
    - Used count-based elevation and smooth colour scaling for readable 3D
      intensity.

14. Streamlit app integration
    - Built app.py as the interactive explorer.
    - Loaded precomputed hex aggregation for the 3D map.
    - Loaded processed row-level Parquet for metrics and raw point display.
    - Added controls for map layer, month, crime type, and raw point limit.
    - Added summary metrics for total rows, filtered rows, map-ready points,
      hex cells, or points drawn.
    - Added dataset details for months, crime types, and police forces.
    - Added graceful empty-state messages when filters produce no results.

15. Current application behaviour
    - Primary layer: 3D hexagons from precomputed visual-ready aggregation.
    - Secondary layer: raw point map for debugging and inspection.
    - Month control supports available months and "All" when multiple months
      exist.
    - Crime type control filters both metrics and active map layer.
    - Heavy binning is done in the pipeline, so Streamlit interactions only
      filter, group, and render smaller datasets.


Current Outcome
---------------
The current version is a reproducible real-data London Crime Pulse Explorer.
It can be regenerated by running:

python run_pipeline.py

Then explored locally with:

streamlit run app.py

The important final architecture is:

raw CSV files
-> inventory and profiling
-> cleaning and validation
-> processed Parquet
-> spatial and hex aggregation
-> visual-ready Parquet
-> Streamlit 3D map app


Possible Later Trimming
-----------------------
When turning this into a final portfolio roadmap, keep the story summary and
only the technical steps that matter to the audience. The most important idea
to preserve is the progression: trust the data first, validate aggregation in
2D, then render the 3D hex map from precomputed data for performance.
